2. Database Schema

7 tables in Supabase PostgreSQL with Row Level Security. All admin access requires authenticated session. Token-based patient access is handled in API routes using the service role key.

patientsUUIDname, phone (UNIQUE), dob, address, aadhaar_last4Last 4 digits of Aadhaar only

content_postsUUIDtitle, body, published, created_atPublic anon read if published=true
settingsINTEGER (always 1)clinic_name, doctor_name, address, phone, email, consent_textSingle row, always id=1
RLS Policies
  • Authenticated (doctor): full access to all tables
  • Anon: read-only on content_posts where published=true
  • Token-based access: handled server-side with service role key — bypasses RLS intentionally
Indexes
  • appointments(token) — fast token lookup
  • appointments(appt_date) — date range queries
  • appointments(patient_id) — patient history
  • patients(phone) — UNIQUE constraint + lookup

4. Key Data Flows

Create Appointment Flow

1. Doctor fills 2-step form (patient phone lookup → appointment details)
2. POST /api/appointments/create (session-authenticated)
3. Service role client: INSERT patients ON CONFLICT (phone) DO UPDATE → get patient_id
4. Generate nanoid(21) token, set token_expires_at = now() + 30 days
5. INSERT appointments (patient_id, date, time, mode, complaint, token)
6. INSERT billing (appointment_id, patient_id, fee=0, items=[])
7. Send SMS via MSG91 with token link to patient phone
8. Return { appointment_id, token_url } → display to doctor with "Resend SMS" option

Patient Token Page Flow

1. Patient opens /appt/[token] (from SMS link)
2. getServerSideProps: calls /api/token/[token] with service role key
3. API validates: token exists AND token_expires_at > now()
4. Returns safe fields only: name, date, time, mode, doctor name, complaint, diagnosis (if completed), prescriptions (if completed), follow_up_date — NEVER returns phone, Aadhaar, billing, clinical notes
5a. If consent_given=false → render ConsentModal (full-page overlay, cannot dismiss by clicking outside)
    — "Yes" → POST /api/consent → hide modal, show full view
    — "No" → hide modal, show minimal view (date, time, clinic name, doctor name only)
5b. If consent_given=true → show full view directly (returning visitor)

5. Security Model

Authentication
  • Doctor: Supabase email OTP (magic link) — no password
  • Account created once manually in Supabase dashboard
  • middleware.ts protects all /admin/* routes — redirects to /login if no session
  • Cron routes verified by Authorization: Bearer CRON_SECRET header
Client Isolation
  • lib/supabase.ts — anon key, browser-safe
  • lib/supabaseAdmin.ts — service role key, API routes ONLY
  • Service role key is NEVER imported in any page or component file
  • HTTPS enforced by Vercel automatically
Data Privacy
  • Aadhaar: last 4 digits only — full number never accepted or stored anywhere
  • Token page: appointment_id never exposed to client
  • Token page: clinical_notes, phone, billing, Aadhaar never returned
  • Patient consent tracked: consent_given + consent_at per appointment
Input Validation
  • Phone: stripped of spaces and leading zeros, stored as 10-digit string
  • All forms: required fields validated client-side AND server-side
  • Aadhaar_last4: CHECK constraint (length = 4) at DB level
  • Appointment status: CHECK constraint for valid enum values
Excluded from MVP: Online payments · Video calls · Multiple staff accounts · Patient self-registration · PDF uploads · WhatsApp · Push/email notifications · Patient-initiated booking
